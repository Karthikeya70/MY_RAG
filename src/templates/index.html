What do these terms mean?

Every piece of jargon on this page, in plain words.

Chunk

The two policy PDFs were cut into {{ chunks }} small pieces, each about a paragraph long, following the document's own section headings. A chunk is one such piece. We search chunks instead of whole documents because a 30-page PDF answers every question at once — a paragraph answers one question well.

Embedding / vector

A way to turn text into a list of 384 numbers that captures its meaning — like coordinates on a giant map of meanings. Texts about similar things (ambulances, hospital transport) land near each other on the map even if they share no words. This project uses the open-source bge-small-en-v1.5 model, running locally, to make these numbers.

Vector database (ChromaDB)

A database that stores all {{ chunks }} embeddings and, given a new one, finds the nearest neighbours fast. This project uses ChromaDB, which runs inside the app like a simple file — no server needed. At larger scale you would switch to pgvector, Qdrant or Pinecone; the idea stays identical.

Retrieval / top-k

The step where your question is embedded and the database returns the k closest chunks (here k = 5). The "Search chunks" button does exactly this and nothing more — it's how you can check what the AI will read before it answers.

Reranker (two-stage retrieval)

The fast search reads your question and each chunk separately — that's what makes it fast, but it misses subtle connections. So we fetch the top 20 candidates fast, then a second model (a cross-encoder) reads your question together with each candidate and re-sorts them by how well they actually answer it. The best 5 survive. Slower per chunk, but it only ever sees 20.

Fact index ("search small, read big")

A fact buried mid-paragraph inside a long multi-topic chunk is nearly invisible to search — the chunk's overall meaning drowns it out. So every prose sentence is also indexed separately, pointing back to its parent chunk. When a single sentence matches your question sharply, its parent chunk gets a vote in the final ranking — and the AI still reads the full parent chunk, so no context is lost. Only the top 5 facts vote: further down, fact matches degrade into lookalikes that would flood the ranking.

Semantic cache

People ask the same thing in different words. Each answered question is stored with its embedding; when a new question lands close enough in meaning (distance ≤ 0.18) and retrieves the same best-matching chunk, the stored answer is reused — no LLM call at all (about 2s instead of 10–15s). Both conditions were set by measurement: "expenses before hospitalisation" vs "after discharge" sound nearly identical (distance 0.106!) but have different answers — the evidence check is what keeps them apart. The cache is per-plan, so a Gold answer can never be served for a Silver question.

Faithfulness (hallucination check)

After the answer is written, a second, independent LLM call audits it: every factual claim in the answer is checked against the retrieved extracts, and anything the model added from its own training knowledge gets flagged. The faithfulness score on the scoreboard = the share of answers with zero unsupported claims. Saying "the document doesn't contain this" counts as faithful — that's exactly the honest behavior we want.

Distance

How far a chunk sits from your question on the meaning map. 0 = same meaning, bigger = less related. Results around 0.15–0.30 are usually on-topic; past 0.4 the match is getting weak. The green bars show closeness (longer bar = closer).

Plan filter

Every chunk is labelled Gold or Silver. Choosing a plan makes the database filter first, search second — so a question about Silver can never accidentally pull a Gold rule. The two plans differ in sneaky ways (Gold has no room-rent limit, Silver caps it at 1% of sum insured per day), so this matters.

Generation / LLM

The final step: the retrieved chunks are pasted into a prompt with strict rules — "answer only from these extracts, cite sections, say so if the answer isn't here" — and a large language model writes the reply. The model is a reading assistant, not a know-it-all: if retrieval fetched the wrong chunks, it cannot secretly fix that.

RAG (Retrieval-Augmented Generation)

The whole recipe above: retrieve the right pieces of a trusted document, then let an LLM generate an answer from them alone. It turns "I think insurance usually covers..." into "Section 4, page 2 says Rs 20,000 per policy year."

hit@5 / answer accuracy

Scores from a 17-question exam this pipeline takes (data/eval) — the scoreboard at the top shows the latest run automatically. hit@5 = the share of questions where the chunk containing the correct evidence appeared in the top 5 results. Answer accuracy = the share of final answers containing the right facts (or correctly refusing when the document has no answer). Rerun python src/evaluate.py --full and refresh this page to update the numbers.
